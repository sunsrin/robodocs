    Glossary¶

    This glossary defines project terms, FRC terms, and logging vocabulary used throughout the documentation.

    Robot Code Terms¶

    Term Meaning
    Robot The WPILib lifecycle class. It configures logging, runs the scheduler, handles mode transitions, and records global telemetry.
    RobotContainer The composition root. It creates subsystems, controllers, autonomous choices, default commands, and button bindings.
    Subsystem A stateful mechanism or service such as drive, launcher, slamtake, hopper, kicker, vision, LEDs, or trash compactor.
    Command A schedulable behavior that uses one or more subsystems. Commands should be non-blocking and declare requirements.
    Requirement The subsystem ownership declared by a command. WPILib uses requirements to interrupt conflicting commands safely.
    Default command The command that runs on a subsystem when no other command requiring that subsystem is scheduled.
    FullSubsystem Project base class for normal subsystems that need periodic input processing and after-scheduler output application.
    VirtualSubsystem Project base class for services that run periodically but are not command requirements, such as vision-style services.
    Goal Desired high-level mechanism state, such as stop, intake, launch, home, or hold position.

    AdvantageKit and Logging¶

    Term Meaning
    AdvantageKit Logging and replay framework used to record inputs, outputs, metadata, and replayable robot behavior.
    AdvantageScope Visualization tool used to inspect WPILOG files, NT4 data, field poses, mechanisms, and time-series values.
    WPILOG Log file format used for recorded robot and replay data.
    NT4 NetworkTables 4, used for live robot telemetry and dashboard communication.
    @AutoLog Annotation used on input classes so AdvantageKit can generate loggable input wrappers.
    Logger.processInputs Logs measured input objects from IO implementations.
    Logger.recordOutput Logs derived values, setpoints, visualization data, command state, and diagnostics.
    @AutoLogOutput Annotation for values computed from object state and logged automatically.
    Replay Desktop workflow where logged inputs from a WPILOG drive robot logic and produce a new _sim log.
    Determinism Property that replaying the same inputs through the same logic should produce the same decisions and outputs.
    LoggedTracer Project utility for recording loop timing checkpoints.

    Hardware Abstraction¶

    Term Meaning
    IO interface Boundary between subsystem logic and hardware or simulator details. Defines logged inputs and requested outputs.
    Inputs Measured facts such as position, velocity, voltage, current, temperature, connection status, or camera frames.
    Outputs Requested actions such as voltage, brake/coast, position setpoint, velocity setpoint, or feedforward.
    No-op IO Implementation that does nothing and reports safe defaults. Used when real hardware is unavailable or during replay paths.
    Real IO Implementation that talks to vendor hardware APIs and configures devices.
    Sim IO Implementation that uses physics models or simulated sensors while preserving the same subsystem logic.
    CAN ID Numeric identifier for a CAN device such as a motor controller, gyro, or encoder.
    CAN bus Physical or logical CAN network. This project documents bus names where they matter.
    Current limit Motor controller configuration that limits electrical current to reduce brownout and hardware damage risk.

    Robot Modes¶

    Term Meaning
    DARWIN Main 2026 robot type. This should be the default merge-safe robot selection.
    ALPHABOT Secondary/test robot type with separate constants where needed.
    SIMBOT Desktop physics simulation robot type selected by setting the SIMBOT environment variable.
    REAL Mode used on the RoboRIO for real robot execution.
    SIM Mode used for desktop physics simulation.
    REPLAY Mode used for desktop log replay when running Darwin or Alphabot off-robot.
    tuningMode Constants flag that enables dashboard-backed tunables. It should be false before merging.

    Vision and Field Terms¶

    Term Meaning
    Northstar Coprocessor-side vision stack that publishes AprilTag and object detection frames to NetworkTables.
    AprilTag layout JSON field layout describing where AprilTags are located for pose estimation.
    Camera transform Robot-relative pose of a camera. Moving cameras need timestamped mechanism state to compute this correctly.
    Timestamped observation Sensor observation associated with the time it was captured, not the time robot code received it.
    Standard deviation Measurement uncertainty passed into pose estimation. Larger values make observations less trusted.
    Object detection Vision output identifying objects such as fuel or robots by class ID and image-space corners.
    tx / ty Horizontal and vertical angular offsets from camera center to a detected target point.
    Fuel Game piece tracked by object detection and simulation.
    Hub shift Project utility for handling shifted hub timing/state information.

    Controls and Operations¶

    Term Meaning
    Primary controller Driver controller on port 0.
    Secondary controller Operator controller on port 1.
    Override switches Physical switch panel on port 5 used for robot-relative drive, coast, and launch-related overrides.
    Field-relative drive Joystick translation is interpreted relative to the field.
    Robot-relative drive Joystick translation is interpreted relative to the robot's current heading.
    Coast override Disabled-mode override that allows selected mechanisms to coast when safe.
    Dashboard chooser Driver Station or AdvantageKit-backed chooser used for auto routines, AprilTag layouts, or questions.

    Autonomous and Trajectories¶

    Term Meaning
    Auto routine Complete autonomous command sequence selected from the dashboard.
    Auto question Dashboard-configurable response that changes how an auto routine is built.
    Choreo Trajectory design/generation tool used for pathing assets.
    VTS Project trajectory generation/deploy format used under src/main/deploy/vts.
    Trajectory reset Pose reset to a trajectory's initial sample. It must be intentional because a bad reset can hide localization issues.
    Alliance flipping Mirroring or transforming field coordinates so the same logic works from either alliance side.

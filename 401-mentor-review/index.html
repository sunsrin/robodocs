    401: Pull Request Review Guide¶

    This guide is for mentors and student leads reviewing robot-code pull requests. Review should focus on behavior, replayability, safety, diagnostics, and maintainability.

    Review Priorities¶

    Review in this order:

    1. Safety and robot damage risk.
    2. Replayability and architecture boundaries.
    3. Runtime behavior and command scheduling.
    4. Simulation/replay coverage.
    5. Logging and dashboard visibility.
    6. Formatting, naming, and cleanup.

    A pretty diff with broken robot semantics is still broken. Start with behavior.

    AdvantageKit IO Pattern¶

    Check:

    • Subsystems do not import vendor hardware classes.
    • Hardware reads happen in updateInputs.
    • Measured values are stored in @AutoLog inputs.
    • Subsystem logic reads from input objects.
    • Output requests are represented as output fields, modes, or setpoint methods.
    • Replay has a no-op path or uses logged inputs.

    Red flags:

    • Calling motor.getPosition() in a subsystem.
    • Reading back desired outputs as if they were measured state.
    • Adding a new sensor value without logging it.
    • Adding real hardware code without a sim or no-op fallback.

    Command-Based Behavior¶

    Check:

    • Commands declare requirements for every subsystem they control.
    • Commands do not block with Thread.sleep() or long loops.
    • Constructor code does not capture state that should be read at schedule time.
    • State reset happens in initialize() or through command factories as appropriate.
    • Default commands do not fight button commands.
    • run, runEnd, startEnd, deadlineFor, raceWith, and finallyDo are used intentionally.

    For button bindings, inspect how triggers compose. Complex trigger chains should still be readable and testable.

    Mode and Constants Safety¶

    Before merge:

    • Constants.robot should be DARWIN.
    • Constants.tuningMode should be false.
    • SIMBOT should not be deployable.
    • Any temporary event, testing, or characterization constants should be removed or isolated.

    checkConstantsPullRequest exists to catch the most important merge mistakes, but reviewers should still read the diff.

    Simulation and Replay¶

    Ask:

    • Can this behavior be checked in SIMBOT?
    • Can this logic be replayed from logs?
    • Does the code behave intentionally when an IO implementation is no-op?
    • Are simulated inputs plausible enough to exercise the subsystem state machine?

    Simulation does not need to be perfect for every mechanism, but it should be honest. A sim class that always reports "at goal" can hide real bugs.

    Logging and Diagnostics¶

    Every meaningful behavior change should leave enough telemetry to debug after the robot leaves the field.

    Check for:

    • Logged inputs for measured values.
    • Logged outputs for setpoints, goals, errors, and selected modes.
    • Alerts for hardware disconnects or unsafe operator selections.
    • Dashboard values only where operators need live feedback.
    • Stable AdvantageScope paths.
    • Units in names when unclear.

    If reviewers cannot tell why a command did or did not run from a log, ask for better telemetry.

    Autonomous Review¶

    For auto changes, check:

    • Path names match files under src/main/deploy/vts.
    • Generated trajectories are committed when expected.
    • Pose reset behavior is intentional.
    • Alliance flipping and mirroring are understood.
    • Auto question responses have sensible defaults.
    • Launch/indexing conditions cannot deadlock silently.
    • Drive and mechanism commands have correct requirements.
    • New dashboard choices are documented.

    Test auto changes in SIMBOT whenever possible.

    Vision Review¶

    For vision changes, check:

    • Camera transforms are in robot coordinates and documented.
    • Moving camera transforms use timestamped mechanism state.
    • Frame parsing matches Northstar output format.
    • Rejection filters are explicit.
    • Standard deviations are plausible.
    • AprilTag layout selection remains logged.
    • Object detections are bounded to field geometry.
    • Disconnected/stale cameras create useful alerts.

    Small sign errors in camera transforms can look plausible, so require AdvantageScope screenshots or logs for calibration-sensitive changes when possible.

    Performance Review¶

    Look for hot-loop problems:

    • New object allocation in periodic() or execute() without a reason.
    • Large stream operations every loop.
    • Network or file IO in periodic code.
    • Reflection in periodic code.
    • Recreating controllers, filters, or profiles every loop instead of only when tunables change.
    • Unbounded lists or maps that grow during a match.

    Use LoggedTracer if the change could affect loop time.

    Hardware Review¶

    For hardware IO additions, check:

    • CAN IDs, bus names, inversion, offsets, and reductions match electrical/mechanical docs.
    • Neutral modes are set intentionally.
    • Current limits are configured.
    • Status signal frequencies are selected intentionally.
    • Faults and connection status are logged.
    • Disabled behavior is safe.
    • Homing and zeroing cannot drive indefinitely at damaging output.

    Real hardware code should fail loudly in logs and alerts when a device is missing.

    Documentation Review¶

    Ask for docs updates when a PR changes:

    • Setup steps.
    • Controls.
    • Dashboard keys.
    • Autonomous routine choices.
    • Subsystem behavior.
    • Hardware IDs, ports, or camera placement.
    • Simulation or replay workflow.
    • Contributor expectations.

    Documentation does not need to describe every line of code, but it should keep operational knowledge out of private memory.

    Final Reviewer Checklist¶

    • Build passes.
    • Spotless has run.
    • Constants are merge-safe.
    • New behavior is logged.
    • Commands have correct requirements.
    • Simulation/replay story is clear.
    • Documentation is updated for user-visible or workflow-visible changes.
    • No unrelated refactors are mixed into the PR.

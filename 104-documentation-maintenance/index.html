    104: Documentation Maintenance¶

    This page explains how to keep the MkDocs site accurate as the robot code changes. Treat documentation like code: it should be reviewed, built, and updated with the behavior it describes.

    Source and Build Output¶

    The documentation source lives in docs_site. MkDocs reads mkdocs.yml and writes generated HTML into site.

    Do not commit site. It is generated output and is published by GitHub Actions.

    python3 -m pip install -r requirements-docs.txt
python3 -m mkdocs serve
python3 -m mkdocs build --strict

    Use mkdocs serve while editing and mkdocs build --strict before opening or merging a pull request.

    When to Update Docs¶

    Update docs in the same pull request when a change affects:

    • First-time setup or required tools.
    • Gradle commands, build behavior, deploy behavior, or CI.
    • Controller bindings, override switches, dashboard keys, or operator alerts.
    • Subsystem behavior, public commands, goals, or safety assumptions.
    • Autonomous routine names, path generation, dashboard questions, or deploy assets.
    • Vision camera IDs, mounts, frame formats, filters, recording behavior, or Northstar topics.
    • Hardware IDs, ports, bus names, offsets, current limits, or device configuration assumptions.
    • Logging paths that someone would use in AdvantageScope.
    • Simulation or replay behavior.

    Small internal refactors do not need a documentation change unless they alter how contributors should work with the code.

    Page Ownership¶

    Use the existing page structure before adding a new page:

    Change type Page to update
    New contributor setup 102-workflow.md
    Driver/operator behavior 103-controls.md
    Architecture or IO boundary changes 201-intermediate.md or 301-architecture.md
    Mechanism package behavior 202-subsystems.md
    Autonomous and trajectory changes 203-auto.md
    Vision or object detection changes 204-vision-object-detection.md
    Simulation or replay changes 205-simulation-replay.md
    Logging paths or dashboard values 302-logging-dashboard.md
    Review expectations 401-mentor-review.md
    Hardware IDs and ports 402-hardware-reference.md
    Repeated implementation patterns Cookbook pages
    Acronyms or project vocabulary glossary.md

    Add a new page only when the topic is large enough that it would make an existing page hard to scan.

    Writing Style¶

    Write for the person doing the work, not for someone already familiar with the code.

    • Start each page with what the reader should use it for.
    • Prefer concrete file names, commands, log keys, and dashboard keys.
    • Explain the operational consequence of a rule, not just the rule.
    • Keep code examples short enough to copy and adapt.
    • Call out safety-critical behavior directly.
    • Use tables for mappings such as controls, constants, dashboard keys, and file ownership.
    • Keep historical notes only when they prevent real confusion.

    Avoid documenting every implementation detail. The docs should explain stable architecture, workflows, and operational knowledge that is easy to lose.

    Links and Navigation¶

    Use relative links between documentation pages:

    [Simulation & Replay](205-simulation-replay.md)

    Use inline code formatting for repository paths, class names, commands, constants, log keys, and dashboard keys.

    When adding a page:

    1. Create the Markdown file under docs_site.
    2. Add it to nav in mkdocs.yml.
    3. Link to it from a relevant existing page.
    4. Run python3 -m mkdocs build --strict.

    Diagrams¶

    MkDocs is configured for Mermaid diagrams. Use diagrams when relationships are easier to understand visually, such as lifecycle flow, data flow, or ownership boundaries.

    Keep diagrams simple. If the diagram needs many labels to be understandable, split it into smaller diagrams or use a table.

    Review Checklist¶

    Before merging documentation changes:

    • The page is listed in mkdocs.yml if it should be user-visible.
    • Links are relative and pass mkdocs build --strict.
    • Commands are copied exactly from the real workflow.
    • Log keys and dashboard keys match the code.
    • Hardware or controls notes match the current robot branch.
    • Generated site output is not committed.
    • Screenshots or external links are necessary and stable.

    Deployment¶

    The check-mkdocs workflow builds docs on pull requests and pushes to main. The deploy-mkdocs workflow publishes the site to GitHub Pages from main.

    Both workflows install the pinned dependencies from requirements-docs.txt, so local builds and CI should use the same MkDocs toolchain.

    202: Subsystems Overview

    This page maps the subsystem packages to the behavior they own. It is a starting point for code navigation and review, not a replacement for reading the source.

    Subsystem Design Pattern

    Most mechanisms follow this shape:

    Layer Responsibility
    Public subsystem class Owns high-level state, commands, logged outputs, alerts, setpoints, and interaction with other robot state.
    IO interface Defines logged inputs and output requests.
    IO implementation Talks to real devices or physics simulation.
    Shared helper Encapsulates repeated logic, such as rollers or swerve modules.

    Subsystems should not know whether they are running real hardware, simulation, or replay.

    Drive

    Path: subsystems/drive

    Main classes:

    • Drive: owns four modules, gyro inputs, kinematics, brake/coast behavior, odometry observations, and drive setpoints.
    • Module: helper for one swerve module.
    • ModuleIO: drive and turn motor input/output boundary.
    • ModuleIOSim: physics simulation for a module using DCMotorSim.
    • GyroIO: gyro input boundary.
    • DriveConstants: robot dimensions, module translations, speed limits, reductions, CAN IDs, encoder offsets, wheel radius, mass, and trajectory constraints.

    Drive.periodic() reads gyro and module inputs, logs them, sends odometry observations to RobotState, updates gyro alerts, and handles disabled brake/coast behavior. Drive.runVelocity() converts chassis speeds to module states, desaturates them, logs setpoints, and passes optimized state requests to each module.

    The module order is front-left, front-right, back-left, back-right.

    RobotState

    Path: RobotState.java

    RobotState is a singleton service rather than a normal motor subsystem. It owns:

    • Odometry pose.
    • Estimated pose after vision fusion.
    • Pose, rotation, and slamtake angle buffers.
    • Robot-relative and field-relative velocity helpers.
    • Vision observation fusion.
    • Intake pose and intake velocity helpers.

    Drive adds odometry observations. Vision adds timestamped AprilTag observations. Slamtake adds slamtake angle observations when enabled and zeroed.

    Slamtake

    Path: subsystems/slamtake

    Slamtake combines a deployable arm (Slam) and an intake roller (RollerSystem). It owns two goal enums:

    • IntakeGoal: INTAKE, OUTTAKE, STOP.
    • SlamGoal: DEPLOY, RETRACT, RETRACT_SLOW, IDLE.

    Important behavior:

    • Intake and outtake voltages are tunable.
    • Deploy and retract are debounced before state changes to DEPLOYED or RETRACTED.
    • Slow retract uses a slew-rate limiter.
    • The measured slamtake angle is recorded into RobotState for timestamped camera pose calculations.
    • The measured angle is visualized in DarwinMechanism3d.

    Hopper

    Path: subsystems/hopper

    Hopper wraps one RollerSystem. Its goals are:

    • LAUNCH.
    • OUTTAKE.
    • STOP.

    It uses tunable voltages for launch and outtake. In SIMBOT mode it can log and update the simulated fuel count through RobotContainer.SimFuelCount.

    Kicker

    Path: subsystems/kicker

    Kicker wraps two rollers: front and back. It owns:

    • Separate PID and feedforward tunables for each roller.
    • Roller radius conversion from launch surface speed to angular velocity.
    • Goals LAUNCH, OUTTAKE, and STOP.
    • SIMBOT launch depletion based on SimFuelCount.launchBPS.

    In periodicAfterScheduler(), it reads LaunchCalculator parameters and converts the target surface speed into front and back roller setpoints.

    Launcher: Flywheel, Hood, and Calculator

    Path: subsystems/launcher

    The launcher is split into several parts:

    • Flywheel: controls the high-energy wheel setpoint with current/power budgeting, feedforward, optional bang-bang behavior, and at-goal checks.
    • Hood: controls launcher angle, homing, force-zeroing, tracking target angle, and 3D visualization.
    • LaunchCalculator: computes launch parameters and presets from robot pose and target geometry.
    • LauncherConstants: launcher geometry used by drive and launch calculations.

    Flywheel limits acceleration based on the configured supply current budget, battery voltage, motor model, efficiency, and flywheel moment of inertia. Hood clamps commands between minAngle and maxAngle after zeroing.

    TrashCompactor

    Path: subsystems/trashcompactor

    TrashCompactor controls a vertical compaction mechanism with a trapezoid profile. Its modes are:

    • PASSIVE_DOWN.
    • FORCE_MAX.
    • FORCE_MIN.
    • LAUNCHING.
    • IDLE.

    It supports homing to min or max, force-zeroing, passive current control, profiled position control, and 3D visualization. Launching mode captures the current measured height plus a tunable offset when the mode is entered.

    RollerSystem

    Path: subsystems/rollers

    RollerSystem is a shared helper used by hopper, kicker, and slamtake. It handles:

    • Logging RollerSystemIOInputs.
    • Motor/follower connection alerts.
    • Battery current reporting.
    • Brake/coast behavior while disabled.
    • Open-loop voltage control.
    • Closed-loop velocity with feedforward.

    Use this helper when adding another simple roller mechanism.

    Vision

    Path: subsystems/vision

    Vision is a VirtualSubsystem, not a motor mechanism. It reads camera data from VisionIO implementations, logs AprilTag and object detection frames, filters bad observations, updates Northstar recording/throttle state, and forwards accepted pose observations to RobotState.

    VisionIONorthstar is the NetworkTables implementation for Northstar. It publishes camera configuration and receives output queues for AprilTags and object detections.

    LEDs

    Path: subsystems/leds

    Leds is a VirtualSubsystem used for driver feedback. RobotContainer sets boolean state such as:

    • superstructureCoast.
    • autoWinnerNotSet.
    • inLaunchingTolerance.

    LedsIOHAL is the HAL-backed IO implementation. A no-op LedsIO is used when LEDs are not wired through a real implementation.

    HubCounter and Hub Shift

    Path: subsystems/hubcounter, util/HubShiftUtil.java

    HubCounter and HubShiftUtil track game-specific hub shift state and override behavior. RobotContainer initializes hub shift state on mode transitions, publishes dashboard outputs, rumbles controllers for missing game data, and lets override switches force the alliance win/loss state when needed.

    201: Architecture & AdvantageKit

    The project is built around AdvantageKit's logged-input model. Subsystems should be deterministic functions of logged inputs, command state, constants, and time. Hardware access is pushed to IO implementations so the same subsystem logic can run on the real robot, in physics simulation, and during log replay.

    Why This Architecture Exists

    FRC robots are difficult to debug if hardware reads, control decisions, and motor writes are mixed together. AdvantageKit solves much of that problem by making inputs explicit and replayable. If the code follows the pattern, a real match log can be replayed on a laptop and the logic should make the same decisions it made on the robot.

    The architecture has three rules:

    1. Inputs from hardware go into @AutoLog input classes.
    2. Subsystems read input objects and write output state.
    3. IO implementations apply the output state to real hardware or simulation after the scheduler.

    IO Pattern

    An IO interface defines the boundary between robot logic and hardware. For example, ModuleIO contains:

    • ModuleIOInputs: logged values read from hardware or simulation.
    • ModuleIOOutputs: desired output mode and setpoints.
    • updateInputs(...): fills the input object.
    • applyOutputs(...): applies the latest desired outputs.
    public interface ModuleIO {
  @AutoLog
  public static class ModuleIOInputs {
    public boolean driveConnected = false;
    public double drivePositionRads = 0.0;
    public double driveVelocityRadsPerSec = 0.0;
    public double driveAppliedVolts = 0.0;
    public boolean turnConnected = false;
    public Rotation2d turnPositionRads = Rotation2d.kZero;
  }

  public static class ModuleIOOutputs {
    public ModuleIOOutputMode mode = ModuleIOOutputMode.COAST;
    public double driveVelocityRadPerSec = 0.0;
    public double driveFeedforward = 0.0;
    public Rotation2d turnRotation = Rotation2d.kZero;
  }

  public default void updateInputs(ModuleIOInputs inputs) {}
  public default void applyOutputs(ModuleIOOutputs outputs) {}
}

    AdvantageKit generates ModuleIOInputsAutoLogged from the @AutoLog class. The subsystem or helper class owns one generated instance and logs it each loop:

    io.updateInputs(inputs);
Logger.processInputs("Drive/Module" + index, inputs);

    Inputs vs Outputs

    Inputs are measured facts. Examples:

    • Encoder position and velocity.
    • Motor applied voltage.
    • Supply or torque current.
    • Temperature.
    • Connection status.
    • Camera frame data.
    • Sensor-derived positions.

    Outputs are requests. Examples:

    • Closed-loop velocity setpoint.
    • Open-loop voltage.
    • Brake or coast mode.
    • Position setpoint.
    • Feedforward value.

    Subsystem logic may use inputs to decide future outputs. It should not use outputs as if they were measured state.

    Periodic and After-Scheduler

    Many subsystems extend FullSubsystem, which supports both normal periodic logic and an after-scheduler hook. The common pattern is:

    1. periodic() reads IO inputs, logs them, updates alerts, computes state, and records telemetry.
    2. WPILib's command scheduler runs commands.
    3. periodicAfterScheduler() applies outputs after commands have had a chance to modify goals.

    The drive module helper is a clear example. Module.periodic() reads/logs inputs and checks alerts. Module.periodicAfterScheduler() calls io.applyOutputs(outputs).

    Robot Lifecycle and Logging

    Robot.java is responsible for global AdvantageKit configuration:

    Mode Data path
    REAL WPILOGWriter and NT4Publisher are added.
    SIM NT4Publisher is added.
    REPLAY WPILOGReader is set as replay source and a _sim WPILOGWriter is added.

    Robot records metadata such as project name, build date, Git SHA, Git branch, dirty state, hostname, and platform. These fields are essential for understanding which code produced a log.

    Determinism in Replay

    Replay is only useful when robot logic does not depend on live hardware calls. During replay:

    • No real motor controller should be needed.
    • Logged input data is the source of sensor truth.
    • The code should write output telemetry into a replay log.
    • Behavior changes can be tested against old logs.

    Avoid hidden dependencies on wall-clock timing, random values, or hardware APIs in subsystem logic. If time is needed, make sure replay behavior is intentional.

    Tunable Numbers

    LoggedTunableNumber is used throughout the code for gains, tolerances, voltages, currents, and timing. These values appear in logs and NetworkTables, making it possible to tune behavior without recompiling in some workflows.

    Common examples:

    • DriveCommands/Launching/kP.
    • DriveTrajectory/LinearkP.
    • Flywheel/kP, kV, kA.
    • Hood/ToleranceDeg.
    • TrashCompactor/Profile/MaxVelocity.
    • ObjectDetection/FuelPersistanceTime.

    When adding a tunable number, choose a stable, descriptive path and document what physical behavior the value affects.

    Alerts

    Subsystems use WPILib Alert objects for hardware faults and operator-visible warnings. Hardware alerts are suppressed in SIM and during the first 30 seconds of non-sim runtime through Robot.showHardwareAlerts() to avoid startup noise.

    Common alert patterns:

    • Debounce connection status before showing an error.
    • Include device role and index in the message.
    • Do not use alerts as a replacement for logged inputs; log the connection state too.

    Current Branch Reality

    This branch currently has simulation implementations for several mechanisms and no-op fallbacks for replay/unimplemented hardware paths. Some real hardware classes are not present. That is acceptable as long as new work preserves the IO boundary. When a real implementation is added, it should implement the existing IO interface rather than changing subsystem logic to import vendor APIs.

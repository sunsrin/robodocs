    301: System Design¶

    This page explains the higher-level design decisions that connect robot lifecycle, subsystem structure, logging, replay, simulation, and operator workflows.

    Composition Root¶

    RobotContainer is the composition root. It owns:

    • Subsystem fields.
    • Controller objects.
    • Override switch triggers.
    • Alerts.
    • Auto selector and AprilTag layout chooser.
    • SIMBOT fuel sim.
    • Default commands.
    • Button bindings.

    This keeps construction and wiring in one place. Subsystem classes should not create controller objects, auto selectors, or unrelated mechanisms.

    Mode Selection¶

    Mode selection starts in Constants:

    public static Mode getMode() {
  return switch (getRobot()) {
    case DARWIN, ALPHABOT -> RobotBase.isReal() ? Mode.REAL : Mode.REPLAY;
    case SIMBOT -> Mode.SIM;
  };
}

    That design intentionally makes desktop Darwin/Alphabot runs use replay. Full physics simulation requires SIMBOT.

    Data Flow Per Loop¶

    The normal loop flow is:

    1. Robot.robotPeriodic() resets cycle timing.
    2. Global battery inputs are logged.
    3. VirtualSubsystem.runAllPeriodic() runs virtual subsystems such as vision and LEDs.
    4. WPILib's CommandScheduler runs.
    5. VirtualSubsystem.runAllPeriodicAfterScheduler() runs virtual after-scheduler hooks.
    6. FullSubsystem.runAllPeriodicAfterScheduler() applies normal subsystem outputs.
    7. Battery logger and global telemetry finish.
    8. Mechanism3d, object detection, hub shift, launch parameters, and dashboard outputs are recorded.

    Subsystems that extend FullSubsystem can split input/logic and output application across the scheduler boundary.

    State Ownership¶

    Use these ownership rules:

    Data Owner
    Hardware sensor readings IO inputs class.
    Mechanism goals Owning subsystem.
    Controller bindings RobotContainer.
    Robot pose estimate RobotState.
    Auto routine selection AutoSelector and RobotContainer.
    Autonomous routine composition AutoBuilder and AutoCommands.
    Vision frame processing Vision.
    Object-derived fuel/robot positions ObjectDetection.
    Driver dashboard output RobotContainer or subsystem closest to the value.

    Avoid duplicate sources of truth. If two classes need the same robot-wide state, it probably belongs in RobotState or a focused utility.

    Hardware Boundary¶

    Subsystems should not import vendor motor-controller or sensor classes. Hardware details belong in IO implementations. The boundary gives the project:

    • Replayable logs.
    • Easier simulation.
    • Cleaner testing.
    • More focused code review.
    • Safer hardware bring-up.

    When a subsystem needs a new measured value, add it to the IO input class and populate it in each implementation.

    Simulation Boundary¶

    Simulation should use the same subsystem logic as the real robot. The simulator should differ only behind IO implementations or in simulation-only services such as FuelSim.

    Good simulation behavior:

    • Responds to applied voltage or setpoint requests.
    • Uses WPILib physics models where practical.
    • Publishes plausible current, position, and velocity.
    • Leaves subsystem logic unchanged.

    Weak simulation behavior:

    • Hardcodes sensor values that do not respond to outputs.
    • Bypasses subsystem state machines.
    • Changes command logic only for simulation.

    Vision and Timestamping¶

    Vision data is timestamped and may arrive later than the current robot loop. RobotState keeps interpolation buffers so AprilTag and object detection observations can be applied at the time the camera frame was captured.

    Important buffers:

    • Pose buffer for odometry pose.
    • Rotation buffer for roll/pitch/yaw.
    • Slam angle buffer for moving camera pose.

    This is why mechanisms that affect camera transforms must publish timestamped observations to RobotState.

    Autonomous Design¶

    Autonomous is designed around reusable command helpers and dashboard questions:

    • AutoSelector handles selection and responses.
    • AutoBuilder converts responses into complete routines.
    • AutoCommands provides reusable primitives.
    • DriveTrajectory follows generated paths.
    • DriveTrajectories and GenerateTrajectories own path generation.

    Do not bury dashboard reads inside low-level commands. Read selected responses through suppliers passed into auto-building code.

    Performance and Loop Timing¶

    The robot loop is configured for 0.02 seconds. The watchdog timeout is widened to 0.2 seconds, and the command scheduler period is set to the same watchdog value. That reduces noise from transient overrun warnings, but it does not make slow code safe.

    Hot paths should avoid:

    • Large allocations every loop.
    • Blocking operations.
    • File or network IO in periodic methods.
    • Repeated expensive stream operations on large data.
    • Long synchronous hardware calls.

    Use LoggedTracer to understand cycle-time impact when changing periodic behavior.

    Design Review Questions¶

    Ask these before merging a substantial change:

    • Does the change preserve replayability?
    • Is each piece of state owned by one class?
    • Are new hardware reads logged as inputs?
    • Are outputs applied after scheduler where appropriate?
    • Is simulation behavior representative enough for the feature?
    • Are dashboard controls logged if they affect behavior?
    • Does the feature fail safely when disconnected, disabled, or unzeroed?
    • Are autonomous and teleop using shared behavior where they should?
    • Can someone debug this from logs after an event?

    Cookbook: Implement Vision Detection

    Use this cookbook when adding or changing AprilTag cameras, object detection cameras, Northstar topics, or robot-side vision filtering.

    1. Identify the Vision Goal

    Decide what the change is meant to provide:

    • Global pose estimation from AprilTags.
    • Fuel detection for autonomous pathing or visualization.
    • Opponent robot detection.
    • Driver or pit diagnostics.
    • Recording or dataset collection.

    The goal determines the required camera pose accuracy, frame rate, exposure, logging, and filtering.

    2. Add or Update CameraConfig

    Edit VisionConstants.cameras. Each CameraConfig needs:

    • poseFunction: robot-to-camera pose as a function of timestamp.
    • id: camera serial or stable identifier published to Northstar.
    • width and height.
    • Exposure, gain, denoise, and color balance.
    • stdDevFactor.
    • fovRads.

    Static camera example:

    CameraConfig.builder()
    .poseFunction(
        timestamp ->
            Optional.of(
                new Pose3d(
                    xMeters,
                    yMeters,
                    zMeters,
                    new Rotation3d(roll, pitch, yaw))))
    .id("camera-id")
    .width(1280)
    .height(960)
    .exposure(4500)
    .gain(5.0)
    .stdDevFactor(1.0)
    .fovRads(Units.degreesToRadians(90.0))
    .build();

    Moving camera example:

    .poseFunction(
    timestamp -> {
      var angle = RobotState.getInstance().getSlamAngle(timestamp);
      if (angle.isEmpty()) {
        return Optional.empty();
      }
      return Optional.of(calculateRobotToCamera(angle.get()));
    })

    If the camera moves, make sure the relevant mechanism writes timestamped state into RobotState.

    3. Configure Northstar Topics

    VisionIONorthstar publishes configuration under:

    northstar_<index>/config

    It subscribes under:

    northstar_<index>/output

    Make sure the coprocessor publishes:

    • observations for AprilTags.
    • objdetect_observations for object detection.
    • fps_apriltags.
    • fps_objdetect.

    The robot publishes timestamp, match info, recording state, layout string, and throttle request. Northstar should consume those fields.

    4. Validate AprilTag Pose Estimation

    In AdvantageScope, inspect:

    • Vision/Inst*.
    • Vision/AprilTags/Inst*.
    • AprilTagVision/Inst*/RobotPose.
    • AprilTagVision/Inst*/TagPoses.
    • AprilTagVision/RobotPoses.
    • RobotState/EstimatedPose.

    Check:

    • Camera pose is on the correct side of the robot.
    • Yaw/pitch/roll signs match WPILib coordinates.
    • Estimated robot pose moves smoothly.
    • Rejected frames are explainable.
    • Latency is plausible.
    • Multi-tag estimates have lower uncertainty than long-distance single-tag estimates.

    5. Tune Filtering

    Filtering lives mostly in VisionConstants and Vision.periodic().

    Adjust carefully:

    • ambiguityThreshold for one-tag pose ambiguity.
    • fieldBorderMargin for off-field rejection.
    • zMin and zMax for impossible robot height.
    • xyStdDevCoefficient and thetaStdDevCoefficient for pose estimator trust.
    • Per-camera stdDevFactor for camera-specific trust.

    Do not tune by feel alone. Compare logs before and after.

    6. Add Object Detection Handling

    Object detection frames are processed in chunks of 10 values:

    class, confidence, tx0, ty0, tx1, ty1, tx2, ty2, tx3, ty3

    The existing robot-side classes are:

    • 0: fuel.
    • 1: robot.

    For a new class, add a new case in Vision.periodic() and define a new observation record or handler in ObjectDetection if needed.

    7. Project tx/ty to Field Coordinates

    ObjectDetection uses:

    • Timestamped robot pose from RobotState.
    • Timestamped robot-to-camera pose from VisionConstants.
    • Camera roll compensation.
    • Camera pitch and target height.
    • Field bounds rejection.
    • Spatial deduplication.

    When adding a target type, define:

    • Which bounding-box points represent the ground contact or physical reference.
    • The expected target height.
    • Maximum persistence time.
    • Deduplication distance.
    • Whether observations should be accepted in teleop, autonomous, sim, or real mode.

    8. Add Diagnostics

    Log enough to debug calibration and filtering:

    • Raw frame arrays through VisionIO inputs.
    • Accepted object translations.
    • Rejected target counts if needed.
    • Camera FPS.
    • Camera latency.
    • Per-camera robot pose.
    • Selected AprilTag layout.

    For a new camera, also document physical mounting and port assignment.

    9. Test Sequence

    1. Verify Northstar connects to NetworkTables.
    2. Verify each camera publishes FPS.
    3. Verify camera config topics match the expected IDs and exposure values.
    4. Point at known AprilTags and inspect robot pose.
    5. Move the robot and confirm pose changes in the correct direction.
    6. Test object detection with known field positions.
    7. Replay logs after any filtering change.
    8. Confirm disconnected-camera alerts behave correctly.

    10. Documentation Updates

    Update these docs when vision changes:

    • 204 Vision & Object Detection for architecture or frame format changes.
    • 402 Hardware Reference Notes for camera IDs, ports, or mounting assumptions.

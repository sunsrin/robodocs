    302: Logging, Dashboard, and Telemetry¶

    This page explains what the code logs, where dashboard outputs come from, and how to add telemetry without breaking replayability.

    AdvantageKit Data Paths¶

    Robot.java configures the data path based on Constants.getMode():

    Mode Receivers/source
    REAL WPILOGWriter and NT4Publisher.
    SIM NT4Publisher.
    REPLAY WPILOGReader source and _sim WPILOGWriter.

    Metadata recorded at startup includes:

    • Project name.
    • Build date.
    • Git SHA.
    • Git date.
    • Git branch.
    • Git dirty state.
    • Hostname.
    • Platform.

    This metadata should be used when comparing logs from different robot builds.

    Input Logging¶

    Use Logger.processInputs(...) for IO input objects generated by @AutoLog. Examples:

    Logger.processInputs("Drive/Gyro", gyroInputs);
Logger.processInputs("Drive/Module" + index, inputs);
Logger.processInputs("Vision/Inst" + i, inputs[i]);
Logger.processInputs("Flywheel", inputs);

    Inputs should represent measured state. If the value comes from a sensor, motor controller status signal, NetworkTables camera frame, or simulated sensor, it belongs in an inputs class.

    Output Logging¶

    Use Logger.recordOutput(...) for derived values, setpoints, visualization objects, and command state. Examples in this project include:

    • SwerveStates/Setpoints.
    • SwerveChassisSpeeds/Measured.
    • DriveTrajectory/Setpoint/Pose.
    • AprilTagVision/RobotPoses.
    • ObjectDetection/FuelTranslations.
    • HubShift/Official.
    • LaunchCalculator/Parameters.
    • CommandsAll/<name>.

    When adding an output, prefer names that group naturally in AdvantageScope.

    AutoLogOutput¶

    Use @AutoLogOutput for values that can be computed from object state and should be logged every loop. Examples:

    • Drive.getModuleStates().
    • Flywheel.atGoal.
    • Hood.getMeasuredAngleRad().
    • TrashCompactor.getMeasuredHeightMeters().
    • RobotState.estimatedPose.

    Do not hide expensive calculations behind @AutoLogOutput. If it allocates heavily or does nontrivial work, consider logging explicitly where the data is already available.

    Command Logging¶

    Robot registers command scheduler callbacks. For every command initialize, finish, or interrupt, it records:

    • CommandsUnique/<name>_<hash> for individual command instances.
    • CommandsAll/<name> for whether any command with that name is active.

    This is useful when debugging conflicts, interrupts, default commands, and button behavior.

    LoggedTracer¶

    LoggedTracer records cycle timing checkpoints. The main loop resets it at the start of robotPeriodic() and records sections such as:

    • BatteryLogger/Periodic.
    • Robot/Commands.
    • Robot/AfterScheduler.
    • ObjectDetection/ClearOldFuelPoses.
    • Drive/Periodic.
    • Vision/Periodic.

    Use it sparingly for runtime sections where loop timing matters.

    SmartDashboard Outputs¶

    SmartDashboard is used for driver-facing outputs and selectors, not as a replacement for logs.

    Examples:

    • Auto routine and question choosers.
    • AprilTag layout chooser.
    • Drive At Goal.
    • Drive Pitch & Roll OK.
    • Match Time.
    • Hub shift status.
    • Hood Angle Offset.
    • Flywheel Speed.
    • Flywheel At Goal.
    • Hood At Goal.
    • Trash Compactor At Min Height.

    If the value is needed for post-match debugging, log it with AdvantageKit too.

    Logged Dashboard Inputs¶

    The code uses AdvantageKit dashboard wrappers such as:

    • LoggedDashboardChooser for auto routines and AprilTag layout selection.
    • LoggedNetworkNumber for /SmartDashboard/Auto/Offset Time?.
    • LoggedNetworkBoolean for /SmartDashboard/Enable Recording.

    These wrappers make operator selections visible in logs.

    Battery and Current Logging¶

    Robot logs battery voltage, RoboRIO current, and Mac Mini current through BatteryLogger. Subsystems report current usage with stable keys, such as:

    • FullDrive/Drive/<index>.
    • FullDrive/Turn/<index>.
    • Flywheel.
    • Hood.
    • TrashCompactor.
    • Roller system names.

    Current usage is used both for telemetry and for understanding performance throttling or brownout risk.

    Naming Guidance¶

    Use stable paths:

    • Good: DriveTrajectory/TranslationError.
    • Good: Flywheel/Feedforward.
    • Good: Vision/ObjDetect/Inst2.
    • Avoid: Debug, Temp, MyValue, or names tied to one temporary experiment.

    Use units in the name when the unit is not obvious:

    • Hood/Profile/GoalVelocityRadPerSec.
    • TrashCompactor/SetpointVelocityMetersPerSec.
    • DriveCommands/Launching/YawToleranceDeg.

    Adding Telemetry Checklist¶

    1. Decide whether the value is an input, derived output, dashboard value, or tunable.
    2. Put measured hardware state in an @AutoLog inputs class.
    3. Put derived state or setpoints under Logger.recordOutput.
    4. Put driver-facing values on SmartDashboard only when drivers or pit crew need them live.
    5. Use LoggedDashboardChooser, LoggedNetworkNumber, or LoggedNetworkBoolean for logged dashboard inputs.
    6. Avoid adding noisy high-frequency arrays unless they are useful in AdvantageScope.
    7. Run simulation or replay and verify the path appears where expected.

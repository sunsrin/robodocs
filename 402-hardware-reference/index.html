    402: Hardware Reference Notes¶

    This page collects hardware-adjacent notes that affect software setup, logging, and debugging. It is not a wiring bible; verify final IDs and wiring against the robot electrical documentation before making hardware changes.

    Robot Selection¶

    Constants.RobotType currently includes:

    • DARWIN.
    • ALPHABOT.
    • SIMBOT.

    The default selected robot is DARWIN. SIMBOT is selected on desktop only when the SIMBOT environment variable is present. Deploying with SIMBOT selected is blocked by checkConstantsDeploy.

    Alphabot Drive Notes¶

    Alphabot constants are also present. Alphabot uses SDS MK4i modules with L3 reduction, a 28 in frame, 34.5 in full width, and 4.69 m/s max linear speed. Keep Alphabot constants intact unless the change is intentionally shared.

    Phoenix 6 CAN Frame Reference¶

    The original Phoenix frame notes in this repository were based only on public Phoenix APIs, not reverse engineering. They were used as a reference for optimizing status signal frequencies.

    Recorded API/device versions:

    • Phoenix API 25.4.0.
    • TalonFX 25.2.1.0.
    • Pigeon 25.5.0.0.
    • CANcoder 25.5.1.0.

    For CAN optimization, prioritize the signals the code actually reads into IO inputs:

    • Drive and turn position.
    • Drive and turn velocity.
    • Applied voltage.
    • Supply and torque current.
    • Temperature.
    • Connection/fault status.
    • Gyro yaw, pitch, and roll.
    • CANcoder absolute/relative position where used.

    Reduce or slow unnecessary status signals on real hardware implementations to avoid bus saturation.

    Hardware Implementation Checklist¶

    When adding a real IO implementation:

    1. Implement the existing IO interface.
    2. Populate every logged input field with real measured data or a clear default.
    3. Apply every output mode intentionally.
    4. Configure neutral mode, current limits, inversion, sensor offsets, PID gains, and feedforward in one place.
    5. Set status signal frequencies based on what the robot actually logs and controls.
    6. Report connection status accurately.
    7. Preserve replay behavior by keeping hardware APIs out of the subsystem class.
    8. Test disabled behavior, enabled behavior, and brownout/current behavior.
    9. Add or update hardware notes when IDs, ports, offsets, or assumptions change.

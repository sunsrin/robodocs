    203: Autonomous & Pathing¶

    Autonomous code combines generated Choreo/VTS trajectories, pose-based drive commands, mechanism commands, and dashboard-selected routine questions. The goal is to make autonomous routines configurable without scattering path names and decision logic throughout the project.

    Important Files¶

    File Purpose
    RobotContainer.java Creates the AutoSelector, registers routines, and returns the selected command to Robot.
    AutoSelector.java Publishes routine choices and up to six routine-specific questions to the dashboard.
    commands/auto/AutoBuilder.java Builds named autonomous routines from smaller commands and selected responses.
    commands/auto/AutoCommands.java Shared auto command helpers: trajectory following, drive-to-pose, indexing, launch helpers, bounds helpers.
    commands/DriveTrajectory.java Follows Choreo SwerveSample trajectories with PID feedback and optional omega override.
    subsystems/drive/DriveTrajectories.java Defines VTS path requests used by trajectory generation.
    util/vts/GenerateTrajectories.java Generates/updates the VTS Choreo project and .traj files.
    src/main/deploy/vts Generated .chor and .traj files deployed to the robot.

    Auto Selection Flow¶

    RobotContainer.configureAutos() creates an AutoBuilder and registers routines with AutoSelector.

    Currently registered routines include:

    • Monopoly Salesman.
    • Timid Salesman.
    • Combobulated Salesman for drive feedforward characterization.
    • Discombobulated Salesman for wheel radius characterization.

    AutoSelector publishes a routine chooser and question choosers under SmartDashboard keys. Each routine can define up to six questions. Responses are stored as AutoQuestionResponse values and read by AutoBuilder suppliers when the command runs.

    Robot.autonomousInit() calls robotContainer.getAutonomousCommand() and schedules the selected command.

    Choreo/VTS Directory¶

    This project sets Choreo's deploy directory to:

    src/main/deploy/vts

    The setup is done reflectively in RobotContainer:

    Choreo.setChoreoDir(new File(Filesystem.getDeployDirectory(), "vts"));

    Use the path names exactly as referenced in code. AutoCommands.followTrajectory("DriveForward1m", ...) expects DriveForward1m.traj to be available in the VTS deploy directory.

    Trajectory Generation¶

    GenerateTrajectories runs after compileJava. It:

    1. Calls Constants.disableHAL() so constants and trajectory generation can run on desktop without normal robot HAL behavior.
    2. Builds a Choreo robot config from DriveConstants.
    3. Ensures src/main/deploy/vts/VTS.chor exists and matches the current robot config hash.
    4. Iterates DriveTrajectories.paths.
    5. Generates or updates .traj files when path requests or robot constants change.

    Running ./gradlew build is the normal way to update generated trajectory files.

    Following a Trajectory¶

    AutoCommands.followTrajectory(...) loads a named Choreo trajectory and optionally resets pose to the trajectory's initial sample.

    DriveTrajectory then:

    • Starts an internal timer.
    • Samples the trajectory at elapsed time.
    • Applies alliance flipping.
    • Optionally applies vertical mirroring.
    • Runs PID feedback on X, Y, and heading.
    • Adds feedforward velocities from the Choreo sample.
    • Sends field-relative chassis speeds to Drive.runVelocity.
    • Logs setpoint pose, setpoint speeds, translation error, rotation error, and whether omega override is active.

    The command ends when its timer reaches the trajectory total time and then stops the drive.

    Pose Resets¶

    A trajectory command can reset odometry at the start. Only reset pose when the robot's actual starting pose is known and matches the trajectory. For chained trajectories, only the first path usually resets pose.

    In timidSalesman, pose reset is selected from the start-position answer before driving and launching. In monopolySalesman, the first selected trajectory starts with pose reset enabled.

    Launching During Auto¶

    Auto launch behavior uses the same launcher infrastructure as teleop:

    • LaunchCalculator supplies flywheel speed, hood angle, kicker surface speed, drive angle, and other launch parameters.
    • AutoCommands.index(...) waits for flywheel readiness, delays for slamtake timing, retracts the slamtake slowly, and runs hopper/kicker launch goals.
    • AutoCommands.followTrajectoryWhileAiming(...) can override trajectory omega with launch-on-the-move aiming output when shouldAim is true.

    This keeps autonomous shooting behavior aligned with teleop shooting behavior.

    Adding a New Auto Routine¶

    Use this checklist:

    1. Add or update the path request in DriveTrajectories.
    2. Run ./gradlew build to generate the .traj file.
    3. Add a helper method to AutoBuilder if the routine is more than a one-liner.
    4. Register the routine in RobotContainer.configureAutos().
    5. Add AutoQuestion entries if the routine needs dashboard-selectable variants.
    6. Use existing helpers such as followTrajectory, driveToPose, index, and returnToClosestLaunchPose.
    7. Test in SIMBOT when possible.
    8. Replay logs or inspect AdvantageScope telemetry for setpoints, pose, and errors.
    9. Update this documentation if the driver station selection flow changes.

    Common Auto Failure Modes¶

    Symptom Likely cause What to check
    Runtime exception: trajectory not found Name mismatch or missing generated file. Compare the string in code to src/main/deploy/vts/*.traj.
    Robot starts path from wrong pose Pose reset is missing or start position answer does not match field setup. Check the selected routine questions and initial sample.
    Path mirrors/flips unexpectedly Alliance flip or vertical mirror path is active. Inspect DriveTrajectory/Mirror and logged trajectory poses.
    Launch command never indexes Flywheel, hood, drive yaw, or tilt tolerance is not satisfied. Inspect Flywheel At Goal, Hood At Goal, Drive At Goal, and pitch/roll dashboard keys.
    Generated files change unexpectedly Drive constants or path request hash changed. Review DriveConstants and DriveTrajectories.

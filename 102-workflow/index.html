    102: Developer Workflow¶

    This page covers the normal development loop: clone, build, format, simulate, replay, deploy, and troubleshoot. All commands should be run from the repository root.

    Prerequisites¶

    Install the normal WPILib Java development environment for the current season, including VS Code and the WPILib extension. This repository uses:

    • Java 17.
    • GradleRIO through the included gradlew wrapper.
    • Spotless with Google Java Format.
    • AdvantageKit and AdvantageScope for logging, replay, and visualization.
    • Vendor dependencies in vendordeps.

    After cloning, install the recommended VS Code extensions when prompted.

    First-Time Setup¶

    1. Open the repository root in VS Code.
    2. Let the WPILib extension detect the Gradle project.
    3. Copy the JSON files from sim into the project root if you want the provided simulation GUI layout and keyboard mapping.
    4. Configure AdvantageScope's custom assets folder if you are using project-specific assets.
    5. Run a build before making changes:
    ./gradlew build

    The build compiles the code, runs configured checks, applies formatting, and generates deploy artifacts.

    Daily Development Loop¶

    Use this loop for most changes:

    1. Pull or update your branch.
    2. Make a small, focused change.
    3. Run formatting or build.
    4. Run simulation or replay if the behavior can be checked without the robot.
    5. Review logs in AdvantageScope when the change affects runtime behavior.
    6. Commit with a clear message.

    Useful commands:

    ./gradlew spotlessApply
./gradlew build
./gradlew simulateJava
./gradlew deploy

    Formatting¶

    Spotless formats Java, Gradle, JSON, Markdown, and .gitignore files. Java uses Google Java Format and the project license header.

    ./gradlew spotlessApply

    compileJava depends on spotlessApply, so ./gradlew build will also format code. Be aware of this when reviewing diffs: formatting changes may appear alongside functional changes if the branch was not already formatted.

    Building¶

    Run:

    ./gradlew build

    Important build behavior:

    • compileJava runs Spotless.
    • compileJava finalizes with generateDriveTrajectories.
    • generateDriveTrajectories creates or updates VTS/Choreo deploy files from DriveTrajectories.
    • The deploy artifact includes files under src/main/deploy.
    • Tests use JUnit 5 when tests are present.

    Constants Checks¶

    Constants.java contains safety checks used by Gradle tasks:

    • checkConstantsDeploy prevents deploying with RobotType.SIMBOT.
    • checkConstantsPullRequest fails if the default robot is not DARWIN or if tuningMode is enabled.

    Before merging, the default constants should be:

    private static final RobotType robot = RobotType.DARWIN;
public static final boolean tuningMode = false;

    Physics Simulation¶

    Use SIMBOT for full physics simulation:

    SIMBOT=1 ./gradlew simulateJava

    In SIMBOT mode, RobotContainer creates simulated IO for drive, slamtake, hopper, kicker, LEDs, and fuel simulation. The simulator also sets the team number to 6328 and uses the WPILib Driver Station simulator.

    The sim GUI is disabled by default in build.gradle to support command-line replay workflows. The Driver Station sim is still added. Copy the files from sim into the project root to use the provided GUI layout and keyboard mappings.

    Log Replay¶

    On desktop without SIMBOT, DARWIN and ALPHABOT run in REPLAY mode. AdvantageKit asks for a log file, reads it through WPILOGReader, and writes a new log with _sim appended to the path.

    Replay is useful when you want to test deterministic logic changes against real match data. Hardware IO should not run during replay; the logged inputs are the data source.

    Documentation Preview¶

    The documentation source lives in docs_site and is built with MkDocs Material. Install pinned documentation dependencies when needed:

    python3 -m pip install -r requirements-docs.txt

    Preview locally:

    python3 -m mkdocs serve

    Run the same strict check used by CI:

    python3 -m mkdocs build --strict

    The generated site directory is build output and should not be committed. See 104 Documentation Maintenance for writing and review expectations.

    Deploying¶

    Connect to the robot network or tether over USB, then run:

    ./gradlew deploy

    Deploy copies the Java artifact and src/main/deploy files to the RoboRIO. checkConstantsDeploy runs before deploy, so deploying with SIMBOT selected fails by design.

    Event Branch Auto-Commits¶

    The eventDeploy Gradle task runs before createVersionFile. If a deploy task is being run and the current Git branch starts with event, the task stages all changes and creates a timestamped commit before deployment.

    This preserves the exact code deployed at an event. It also means a deploy from an event... branch can create a commit even if you did not explicitly run git commit.

    Common Troubleshooting¶

    Symptom Likely cause What to check
    Desktop run asks for a replay log You are running DARWIN or ALPHABOT on desktop. Use SIMBOT=1 ./gradlew simulateJava for physics sim.
    Deploy fails with invalid robot selected Constants.robot is SIMBOT. Restore DARWIN or ALPHABOT before deploying.
    Pull request check fails constants validation tuningMode is true or robot type is not default. Restore merge-safe constants.
    Choreo trajectory not found File missing under src/main/deploy/vts or name mismatch in code. Run ./gradlew build and check DriveTrajectories path names.
    Controller inputs look wrong in sim Simulation GUI mappings were not copied. Copy sim/*.json to the repository root and review sim/Simulation Key Mapping.html.
    AdvantageScope data is missing Wrong mode or no receiver configured for your workflow. REAL logs WPILOG+NT4, SIM publishes NT4, REPLAY writes a replay log.

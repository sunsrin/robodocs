    Cookbook: Add a New Subsystem

    Use this cookbook when adding a mechanism that should follow the AdvantageKit IO pattern. The example uses an elevator, but the same structure applies to rollers, pivots, climbers, and other mechanisms.

    1. Decide the Ownership Boundary

    Before creating files, decide:

    • What physical mechanism or service owns the state?
    • What measured values are required?
    • What commands or goals will the subsystem expose?
    • What outputs must be applied to hardware?
    • What should happen when disabled, unzeroed, disconnected, or in replay?
    • What simulation fidelity is needed?

    If two mechanisms must always act together, they may belong in one subsystem. If they have separate state, separate hardware, and separate commands, keep them separate.

    2. Create the IO Interface

    Create an interface in the subsystem package:

    public interface ElevatorIO {
  @AutoLog
  public static class ElevatorIOInputs {
    public boolean connected = false;
    public double positionMeters = 0.0;
    public double velocityMetersPerSec = 0.0;
    public double appliedVolts = 0.0;
    public double supplyCurrentAmps = 0.0;
    public double tempCelsius = 0.0;
  }

  public static enum ElevatorIOOutputMode {
    COAST,
    BRAKE,
    OPEN_LOOP,
    CLOSED_LOOP
  }

  public static class ElevatorIOOutputs {
    public ElevatorIOOutputMode mode = ElevatorIOOutputMode.BRAKE;
    public double appliedVolts = 0.0;
    public double positionMeters = 0.0;
    public double velocityMetersPerSec = 0.0;
    public double feedforwardVolts = 0.0;
    public double kP = 0.0;
    public double kD = 0.0;
  }

  public default void updateInputs(ElevatorIOInputs inputs) {}

  public default void applyOutputs(ElevatorIOOutputs outputs) {}
}

    Use inputs for measured state only. Use outputs for requests.

    3. Create the Subsystem

    The subsystem owns the IO object, generated inputs, outputs, goals, alerts, and commands.

    public class Elevator extends FullSubsystem {
  private final ElevatorIO io;
  private final ElevatorIOInputsAutoLogged inputs = new ElevatorIOInputsAutoLogged();
  private final ElevatorIOOutputs outputs = new ElevatorIOOutputs();

  private final Alert disconnected =
      new Alert("Elevator motor disconnected!", Alert.AlertType.kError);

  private double goalMeters = 0.0;

  public Elevator(ElevatorIO io) {
    this.io = io;
  }

  @Override
  public void periodic() {
    io.updateInputs(inputs);
    Logger.processInputs("Elevator", inputs);

    disconnected.set(Robot.showHardwareAlerts() && !inputs.connected);

    if (DriverStation.isDisabled()) {
      outputs.mode = ElevatorIOOutputMode.BRAKE;
    }
  }

  @Override
  public void periodicAfterScheduler() {
    if (DriverStation.isEnabled()) {
      outputs.mode = ElevatorIOOutputMode.CLOSED_LOOP;
      outputs.positionMeters = goalMeters;
    }

    io.applyOutputs(outputs);
  }

  public Command runToHeight(DoubleSupplier meters) {
    return run(() -> goalMeters = meters.getAsDouble());
  }
}

    Use FullSubsystem when the subsystem needs after-scheduler output application. Use VirtualSubsystem for non-command-required services such as vision or LEDs.

    4. Add Simulation IO

    Use WPILib simulation classes where possible.

    public class ElevatorIOSim implements ElevatorIO {
  private final ElevatorSim sim = new ElevatorSim(...);
  private double appliedVolts = 0.0;

  @Override
  public void updateInputs(ElevatorIOInputs inputs) {
    sim.setInputVoltage(MathUtil.clamp(appliedVolts, -12.0, 12.0));
    sim.update(Constants.loopPeriodSecs);

    inputs.connected = true;
    inputs.positionMeters = sim.getPositionMeters();
    inputs.velocityMetersPerSec = sim.getVelocityMetersPerSecond();
    inputs.appliedVolts = appliedVolts;
    inputs.supplyCurrentAmps = Math.abs(sim.getCurrentDrawAmps());
  }

  @Override
  public void applyOutputs(ElevatorIOOutputs outputs) {
    switch (outputs.mode) {
      case COAST, BRAKE -> appliedVolts = 0.0;
      case OPEN_LOOP -> appliedVolts = outputs.appliedVolts;
      case CLOSED_LOOP -> {
        // Run a simulated controller here or use a simple profiled model.
      }
    }
  }
}

    Simulation should react to output requests. Avoid returning fixed sensor values unless the mechanism truly cannot move in the test.

    5. Add Real IO

    When hardware is ready, create a real implementation such as ElevatorIOTalonFX or ElevatorIOSparkMax.

    Real IO should:

    • Configure motor controllers in the constructor.
    • Apply current limits and neutral modes.
    • Set inversion and sensor conversion factors.
    • Set status signal rates.
    • Read all measured values in updateInputs.
    • Apply output modes in applyOutputs.
    • Report connection status accurately.
    • Avoid subsystem-specific decisions.

    If real hardware is not ready, leave the subsystem wired to no-op IO for that robot path and document the gap.

    6. Wire It in RobotContainer

    Instantiate the subsystem based on robot/mode. Follow existing patterns and preserve no-op replay behavior.

    switch (Constants.getRobot()) {
  case SIMBOT -> elevator = new Elevator(new ElevatorIOSim());
  case DARWIN -> {
    // Use real IO when implemented.
  }
  case ALPHABOT -> {
    // Use real or no-op IO as appropriate.
  }
}

if (elevator == null) {
  elevator = new Elevator(new ElevatorIO() {});
}

    Then configure:

    • Default command.
    • Button bindings.
    • Auto usage.
    • Dashboard outputs.
    • Coast overrides if applicable.

    7. Add Commands

    Prefer command factory methods on the subsystem or in a focused command utility class. Commands should declare requirements by using subsystem methods such as run, runEnd, or by passing the subsystem to Commands.run.

    public Command holdHome() {
  return run(() -> goalMeters = 0.0);
}

public Command runOpenLoop(DoubleSupplier volts) {
  return runEnd(
      () -> {
        outputs.mode = ElevatorIOOutputMode.OPEN_LOOP;
        outputs.appliedVolts = volts.getAsDouble();
      },
      () -> outputs.mode = ElevatorIOOutputMode.BRAKE);
}

    Do not block in commands.

    8. Add Logging and Alerts

    At minimum, log:

    • Inputs through Logger.processInputs.
    • Goal state.
    • Setpoint state.
    • Measured state if not already obvious from inputs.
    • At-goal booleans.
    • Mode enums.

    Add alerts for disconnected hardware and unsafe configuration.

    9. Test

    Use this sequence:

    1. ./gradlew build.
    2. SIMBOT=1 ./gradlew simulateJava if simulation is wired.
    3. Replay a log if the behavior can be checked from logged inputs.
    4. Confirm telemetry appears in AdvantageScope.
    5. Test disabled behavior before enabled behavior on hardware.
    6. Test homing/zeroing at low risk before closed-loop movement.

    Review Checklist

    • IO boundary is clean.
    • Inputs are logged.
    • Outputs are applied intentionally.
    • No vendor classes in the subsystem.
    • Simulation or no-op behavior is present.
    • Commands declare requirements.
    • Disabled behavior is safe.
    • Hardware faults create useful alerts.
    • Documentation and controls pages are updated if behavior is operator-visible.

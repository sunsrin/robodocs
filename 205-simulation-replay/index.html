    205: Simulation & Replay¶

    Simulation and replay are separate workflows. They both run on desktop, but they answer different questions.

    Workflow Robot type/mode Use it for
    Physics simulation SIMBOT / SIM Testing commands, mechanism logic, fuel sim, basic driver behavior, and dashboard outputs without a robot.
    Log replay DARWIN or ALPHABOT on desktop / REPLAY Testing deterministic logic changes against recorded real or simulated logs.
    Real robot DARWIN or ALPHABOT on RoboRIO / REAL Practice, tuning, competition, and hardware validation.

    Running Physics Simulation¶

    Run:

    SIMBOT=1 ./gradlew simulateJava

    Constants.getRobot() selects SIMBOT on desktop only when the SIMBOT environment variable is present. Constants.getMode() then returns SIM.

    SIMBOT currently creates:

    • Drive with four ModuleIOSim instances and no-op gyro IO.
    • Slamtake with SlamIOSim and RollerSystemIOSim.
    • Hopper with RollerSystemIOSim and simulated fuel count.
    • Kicker with no-op roller IO for the current branch plus simulated fuel count.
    • Leds with LedsIOHAL.
    • FuelSim for field fuel state.

    Any subsystem not explicitly created for SIMBOT falls back to a no-op IO implementation.

    Simulation GUI¶

    The sim directory contains:

    • simgui.json.
    • simgui-ds.json.
    • simgui-window.json.
    • Simulation Key Mapping.html.

    Copy the JSON files into the project root to use the provided WPILib simulation GUI layout. Open the HTML file to see keyboard mappings for the simulated Xbox and override controllers.

    The sim GUI is disabled by default in build.gradle:

    wpi.sim.addGui().defaultEnabled = false
wpi.sim.addDriverstation()

    This keeps command-line replay workflows cleaner while still enabling the Driver Station simulator.

    FuelSim¶

    RobotContainer.configureFuelSim() registers:

    • The robot footprint.
    • The intake footprint.
    • A predicate for when intake is active.
    • A callback for increasing stored fuel.

    SIMBOT starts with 8 fuel stored in SimFuelCount. The simulated capacity is 80, and the simulated launch rate is 16 fuel per second. In autonomous, fuel sim clears and respawns starting fuel, then resets stored fuel to 8.

    ObjectDetection.getFuelTranslations() returns FuelSim fuel positions in SIMBOT mode.

    Running Replay¶

    Run a desktop simulation without SIMBOT:

    ./gradlew simulateJava

    Because the default robot is DARWIN, desktop execution uses REPLAY. AdvantageKit finds a replay log, reads it, and writes a new log with _sim appended.

    Replay should not require real hardware IO. RobotContainer creates no-op implementations for any subsystem that was not created by the selected robot path.

    What Replay Can Catch¶

    Replay is useful for:

    • Checking whether a logic change would have changed an old match.
    • Testing vision filtering changes.
    • Reviewing auto or drive command behavior against recorded pose data.
    • Verifying logged outputs and dashboard calculations.
    • Debugging state-machine behavior without the robot.

    Replay is not enough for:

    • Validating motor controller configuration.
    • Validating CAN IDs or wiring.
    • Measuring real mechanism response.
    • Tuning feedforward values that depend on real hardware.
    • Checking camera exposure or physical calibration.

    AdvantageScope Signals to Inspect¶

    When testing simulation or replay, inspect:

    Signal Why it matters
    SwerveStates/Measured Whether simulated or replayed modules report believable state.
    SwerveStates/Setpoints Whether commands are requesting the expected module states.
    RobotState/EstimatedPose Main pose estimate after odometry and vision fusion.
    AprilTagVision/RobotPoses All accepted AprilTag-derived robot poses.
    DriveTrajectory/TranslationError Path-following error.
    DriveTrajectory/RotationError Heading error while following paths.
    Flywheel/Setpoint and Flywheel/Goal Whether launcher control is tracking the desired target.
    Hood/Profile/GoalPositionRad Hood target angle.
    ObjectDetection/FuelTranslations Simulated or detected fuel positions.
    CommandsAll/* Which command names are active.
    Throttled Whether disabled throttling is active.

    Replay-Safe Code Checklist¶

    Before merging subsystem or command changes, check:

    • Hardware reads happen only in updateInputs.
    • Subsystem logic uses the inputs object, not vendor classes.
    • Outputs are applied in the IO layer.
    • Any new input field is included in an @AutoLog inputs class.
    • New commands declare subsystem requirements.
    • New runtime decisions record enough telemetry to debug later.
    • Sim and replay behavior are intentional when hardware IO is missing.

    Troubleshooting Simulation and Replay¶

    Symptom Likely cause Fix
    Simulation launches replay file chooser SIMBOT environment variable is missing. Run SIMBOT=1 ./gradlew simulateJava.
    No mechanism moves in SIMBOT The subsystem may be using no-op IO. Check RobotContainer for a SIMBOT implementation.
    Logs are missing live NT4 data in replay Replay writes a WPILOG output instead of publishing all live data. Open the generated _sim log in AdvantageScope.
    Fuel count does not change Intake predicate or launch goal is not active. Check slamtake goals, kicker goal, and Hopper/SimFueldCount.
    Driver controls do nothing in sim Sim DS/controller mappings may not be configured. Copy the sim JSON files and use the HTML key map.

    Hopper/SimFueldCount is the current logged key in code, including the spelling. If the code is renamed later, update saved layouts and this troubleshooting table together.

    204: Vision & Object Detection¶

    The vision stack combines Northstar coprocessors, AprilTag pose estimation, object detection frames, timestamped robot state, and AdvantageKit logging. Vision does not command motors directly; it publishes observations that other systems can use.

    Main Files¶

    File Purpose
    subsystems/vision/Vision.java Reads all camera IO, logs frames, filters AprilTag observations, handles object detections, and forwards accepted pose observations.
    subsystems/vision/VisionIO.java Defines NetworkTables connection status, AprilTag frame inputs, object detection frame inputs, recording control, and throttle control.
    subsystems/vision/VisionIONorthstar.java NetworkTables implementation for Northstar devices.
    subsystems/vision/VisionConstants.java Camera IDs, resolutions, exposure/gain, robot-to-camera transforms, filtering constants, and confidence thresholds.
    ObjectDetection.java Converts object detection tx/ty observations into field-relative fuel and robot translations.
    RobotState.java Provides timestamped pose, rotation, and slamtake-angle buffers used by vision.
    northstar Coprocessor-side workers and support code.

    Northstar Instances¶

    VisionIONorthstar creates one NetworkTables table per device:

    northstar_0
northstar_1
northstar_2
northstar_3

    For each instance, robot code publishes camera configuration under config and subscribes to frame output under output.

    Published configuration includes:

    • Camera ID.
    • Resolution.
    • Exposure, gain, denoise.
    • Color balance values.
    • AprilTag fiducial size.
    • Recording state.
    • Timestamp.
    • Event name, match type, and match number.
    • AprilTag layout.
    • Throttle FPS.

    Subscribed outputs include:

    • observations: AprilTag frames.
    • objdetect_observations: object detection frames.
    • fps_apriltags.
    • fps_objdetect.

    Camera Configuration¶

    VisionConstants.cameras is selected by robot type.

    Darwin currently defines four cameras:

    Index Camera ID Resolution Main use
    0 40552081 1800x1200 Monochrome AprilTag camera facing rearward.
    1 24737133 1280x960 Color camera with a moving pose based on slamtake angle.
    2 24736167 1280x960 Color camera angled to one side.
    3 24736221 1280x960 Color camera angled to the other side.

    Alphabot currently defines two cameras.

    Camera 1 on Darwin is special: its pose depends on RobotState.getSlamAngle(timestamp). If the slamtake angle is not available for the frame timestamp, that camera returns Optional.empty() and the frame is skipped.

    AprilTag Frame Processing¶

    Vision.periodic() reads all queued frames from each camera. For each AprilTag frame:

    1. It gets the frame timestamp.
    2. It gets the camera pose at that timestamp.
    3. It skips blank frames.
    4. It decodes either a multi-tag single-pose result or a one-tag two-pose ambiguous result.
    5. For ambiguous one-tag results, it chooses the pose whose rotation is closer to the current estimated robot rotation if the ambiguity ratio is acceptable.
    6. It rejects poses outside the field border margin.
    7. It rejects poses outside allowed Z bounds.
    8. It finds tag poses in the selected AprilTag layout.
    9. It calculates average camera-to-tag distance.
    10. It computes XY and theta standard deviations.
    11. It creates a RobotState.VisionObservation.

    Accepted observations are sorted by timestamp and sent to:

    RobotState.getInstance().addVisionObservation(...)

    Vision Filtering Constants¶

    Important constants in VisionConstants:

    Constant Meaning
    ambiguityThreshold Used to reject or choose between ambiguous one-tag pose solutions.
    fieldBorderMargin Allowed margin outside field before rejecting a robot pose.
    zMin, zMax Allowed robot Z range for accepted vision poses.
    xyStdDevCoefficient Scales XY measurement uncertainty.
    thetaStdDevCoefficient Scales rotation measurement uncertainty.
    targetLogTimeSecs How long recently detected tag poses remain visible in logs.

    The standard deviation model increases uncertainty with distance and decreases uncertainty with more detected tags.

    AprilTag Layout Selection¶

    RobotContainer creates a logged dashboard chooser named AprilTag Layout. Options include:

    • Official.
    • Hub.
    • Tower.
    • Outpost.
    • None.

    The selected layout is passed to Vision and published to Northstar. RobotContainer raises an alert when a non-default layout is selected.

    AprilTag layout JSON files live under:

    src/main/deploy/apriltags

    They are grouped by layout source/type such as andymark, hq, and welded.

    Object Detection Frames¶

    Object detection frames are arrays of detections. Each detection uses 10 values:

    1. Class ID.
    2. Confidence.
    3. Corner 0 tx.
    4. Corner 0 ty.
    5. Corner 1 tx.
    6. Corner 1 ty.
    7. Corner 2 tx.
    8. Corner 2 ty.
    9. Corner 3 tx.
    10. Corner 3 ty.

    The current class IDs are:

    • 0: fuel.
    • 1: robot.

    Northstar enforces the confidence threshold, so the robot-side thresholds currently default to 0.0.

    Fuel Position Tracking¶

    ObjectDetection stores fuel positions in a spatial hash grid with 1-meter cells. The flow is:

    1. Autonomous initializes assumed starting fuel positions.
    2. Vision clears old fuel and fuel inside the robot intake footprint.
    3. For the most recent object detection frame from a camera, vision clears fuel in that camera's field of view.
    4. New fuel observations are projected to the field using robot pose, camera pose, and corner tx/ty data.
    5. Fuel outside the field is rejected.
    6. Nearby fuel records are deduplicated using a tunable overlap distance.

    In real mode, fuel observations are ignored outside autonomous. In SIMBOT mode, ObjectDetection.getFuelTranslations() returns fuel from FuelSim.

    Robot Position Tracking¶

    Robot detections use the lower bounding-box corners to estimate the top-down position of another robot. They are projected through the same timestamped pose and camera-pose process, then deduplicated using ObjectDetection/RobotOverlap.

    Recent robot translations are logged by Robot.robotPeriodic() as 3D poses for visualization.

    Recording and Throttling¶

    Vision publishes recording and throttle requests to Northstar:

    • Recording is enabled when FMS has been attached long enough, when /SmartDashboard/Enable Recording is true, or when alwaysRecord is true.
    • Throttle is set to 1.0 FPS when Robot.shouldThrottle() returns true.
    • Otherwise throttle is set to -1.0, meaning no requested throttle.

    Robot.shouldThrottle() becomes true after the robot has been disabled for 5 seconds.

    Disconnect Alerts¶

    Each Northstar instance has a disconnect timer. Alerts distinguish between:

    • Northstar disconnected from NetworkTables.
    • Northstar connected to NetworkTables but not publishing frames.

    If no AprilTag or object detection frames arrive for 1.5 seconds, the instance is treated as disconnected or stale.

    Adding or Moving a Camera¶

    Use this checklist:

    1. Add or update the CameraConfig in VisionConstants.
    2. Set the correct camera ID, resolution, exposure, gain, denoise, color balance, FOV, and stdDevFactor.
    3. Define the camera pose as a function of timestamp.
    4. If the camera moves, ensure the required mechanism state is buffered in RobotState.
    5. Update Northstar device setup so the expected camera ID is attached to the expected instance.
    6. Test NetworkTables connectivity and frame publishing.
    7. Check AprilTagVision/Inst*/RobotPose, TagPoses, latency, and FPS in AdvantageScope.
    8. Update this page and any hardware port notes.

    101: Welcome to FRC Software¶

    This page introduces the structure of the 2026 robot code and gives new contributors enough context to open the project without getting lost. The code is Java, uses WPILib's command-based framework, and is organized around AdvantageKit replayability.

    The Big Picture¶

    Robot code runs in a fixed loop. In this project the loop period is 0.02 seconds, so the main control loop runs about 50 times per second. Each loop:

    1. Reads hardware inputs through IO classes.
    2. Logs those inputs through AdvantageKit.
    3. Runs subsystem periodic methods and scheduled commands.
    4. Computes new goals or outputs.
    5. Applies outputs to hardware or simulation.

    That order matters. A subsystem should make decisions from the latest input snapshot, not by reaching directly into a motor controller whenever it wants information.

    Command-Based Programming¶

    WPILib command-based programming splits robot software into a few recurring pieces:

    Piece What it means in this repo
    Subsystem A robot mechanism or stateful service such as Drive, Slamtake, Hopper, Kicker, Hood, Flywheel, TrashCompactor, Vision, or Leds.
    Command A behavior that uses one or more subsystems, such as joystick drive, launch alignment, indexing, homing, or following a trajectory.
    RobotContainer The composition root. It creates subsystems, creates controller objects, configures autonomous choices, binds buttons to commands, and sets default commands.
    Robot The lifecycle owner. It starts logging, runs the scheduler, handles mode transitions, and records global telemetry.

    Subsystems own mechanism state. Commands ask subsystems to do things. Controllers and dashboard choices schedule commands.

    Startup Flow¶

    The startup path is:

    1. Main.java starts WPILib's robot runtime.
    2. Robot.java constructs the LoggedRobot.
    3. Robot records build metadata and configures AdvantageKit data receivers.
    4. Robot creates RobotContainer.
    5. RobotContainer creates subsystems, controllers, auto routines, default commands, and button bindings.
    6. WPILib calls robotPeriodic() every loop, which runs the command scheduler and the project's after-scheduler hooks.

    When debugging startup behavior, begin with Constants.java, then Robot.java, then RobotContainer.java.

    Robot Modes¶

    Constants.getMode() determines how the code talks to the outside world:

    Mode When it is used What happens
    REAL Running on the RoboRIO with DARWIN or ALPHABOT selected. Logs to WPILOG and publishes live NT4 telemetry.
    SIM Running on desktop with the SIMBOT robot selected. Uses physics simulation IO implementations and publishes NT4 telemetry.
    REPLAY Running on desktop with DARWIN or ALPHABOT selected. Reads an existing WPILOG and writes a replay output log.

    For physics simulation, run with SIMBOT selected through the environment:

    SIMBOT=1 ./gradlew simulateJava

    Without SIMBOT, desktop execution is intended for replay, not full physics simulation.

    Subsystems and IO¶

    Most mechanisms are split into:

    • A subsystem class, such as Drive.java or Flywheel.java.
    • An IO interface, such as ModuleIO.java or FlywheelIO.java.
    • One or more IO implementations, such as ModuleIOSim.java.

    The subsystem contains robot logic. The IO implementation contains hardware or simulator details. If a subsystem imports vendor hardware classes directly, the code becomes harder to replay and test.

    A Minimal Command¶

    A command should declare which subsystem it uses and should return quickly every loop. In this project many simple commands are created with Commands.run, runEnd, startEnd, or subsystem helper methods.

    public static Command stopDrive(Drive drive) {
  return Commands.run(drive::stop, drive);
}

    The final drive argument is the requirement. It tells WPILib that this command owns the drive subsystem while scheduled, so conflicting drive commands are interrupted correctly.

    What to Read First¶

    Start with these files:

    • Constants.java: robot selection, loop timing, tuning flag, deploy checks.
    • Robot.java: logging setup, scheduler execution, mode transitions, global telemetry.
    • RobotContainer.java: subsystem creation, button bindings, auto setup.
    • subsystems/drive/Drive.java: representative subsystem structure.
    • subsystems/drive/ModuleIO.java: representative IO interface with logged inputs and output state.
    • commands/DriveCommands.java: examples of command factory methods.
    • commands/auto/AutoBuilder.java: examples of combining smaller commands into autonomous routines.
